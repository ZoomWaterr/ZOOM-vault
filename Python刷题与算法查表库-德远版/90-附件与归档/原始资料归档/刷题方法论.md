# 🧭 一、通用解题框架（5 步走）

```
┌──────────────────────────────────────────────────────────────────┐
│                                                                  │
│  STEP 1 ── 读懂题意                                               │
│  ├─ 用自己的大白话复述一遍：这题到底要我干嘛？                      │
│  ├─ 圈出关键数字：n 最大多少？有没有负数？能不能整除？              │
│  └─ 如果题目给了样例，先手算一遍，确认自己理解了                    │
│                                                                  │
│  STEP 2 ── 建立模型                                               │
│  ├─ 判断题型：这是求最值？计数？判断可行？输出方案？                │
│  └─ 把题目的具体描述，抽象成「输入 → 处理 → 输出」                 │
│                                                                  │
│  STEP 3 ── 匹配算法  ← 翻下面的「题型模式识别表」                   │
│  ├─ 看到"连续子数组" → 滑动窗口 / 前缀和                           │
│  ├─ 看到"所有方案"   → 回溯 / DFS                                 │
│  ├─ 看到"最短路径"   → BFS                                        │
│  └─ 拿不准 → 看 n 的范围倒推 → 翻输入规模对照表                     │
│                                                                  │
│  STEP 4 ── 写代码                                                 │
│  ├─ 先搭框架：def solve() → 输入处理 → 核心逻辑 → 输出             │
│  ├─ 核心逻辑直接打开 算法模板库.md，复制模板改                      │
│  └─ 写完主逻辑再加边界处理                                         │
│                                                                  │
│  STEP 5 ── 验证 + 调试                                            │
│  ├─ 先跑题目样例 → 过了才提交                                      │
│  ├─ 没过？跑「边界条件检查清单」（下面有）                           │
│  └─ 还是没过？→ 跳到「调试技巧」                                    │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

**举个完整例子**——题目："给一个数组和整数 K，求和为 K 的连续子数组有几个"

| 步骤 | 具体思考 |
|:---|:---|
| Step 1 | n ≤ 20000，可能有负数，求**个数**（不是求最长/最短） |
| Step 2 | "连续子数组" + "和为 K" → 子数组和 = 前缀和之差。问题变成：有多少对前缀和之差 = K |
| Step 3 | "前缀和 + 统计" → 翻算法模板库 Part 2 + Part 9（哈希表计数） |
| Step 4 | 遍历一遍，用字典记录每个前缀和出现了几次，同时查 `当前前缀和 - K` 出现次数累加 |
| Step 5 | 试 `[1,1,1] K=2` → 答案是 2 吗？试 `K=0` 空数组？试 n=1 单元素？ |

## 按"n 的取值范围"倒推（题目问法模糊时的救命稻草）

> 题目可能用词模糊，但数据范围不会骗人。

| n 最大多少 | 你能用的算法 | 绝对不能用的 |
|:---:|:---|:---|
| ≤ 10 | 全排列、暴力枚举、$O(n!)$ | — |
| ≤ 20 | 状态压缩 DP、子集生成 $O(2^n)$ | — |
| ≤ 100 | 三重循环 $O(n^3)$、Floyd、区间 DP | $O(2^n)$ |
| ≤ 500 | $O(n^3)$ 勉强能过 | $O(2^n)$ |
| ≤ 2,000 | 双重循环 $O(n^2)$、冒泡 | $O(n^3)$ 危险 |
| ≤ 10⁵ | 排序 $O(n \log n)$、堆、二分 | $O(n^2)$ 必超时 |
| ≤ 10⁶ | 单次遍历 $O(n)$、前缀和 | $O(n \log n)$ 可能卡 |
| ≥ 10⁷ | $O(n)$ 或更优，一次扫描 | 多一层循环都危险 |

> **速算**：Python 约 $10^7$ 次操作/秒。$n=10^5$ 时 $O(n^2)=10^{10}$，必超。

| 数据量 n | $O(n)$ | $O(n \log n)$ | $O(n^2)$ | $O(2^n)$ |
|:---:|:---:|:---:|:---:|:---:|
| 100 | ✅ 瞬间 | ✅ 瞬间 | ✅ 瞬间 | ❌ 宇宙毁灭 |
| 1,000 | ✅ 瞬间 | ✅ 瞬间 | ✅ ~0.01s | ❌ |
| 10,000 | ✅ 瞬间 | ✅ 瞬间 | ⚠️ ~0.5s | ❌ |
| 100,000 | ✅ 瞬间 | ✅ ~0.02s | ❌ **超时** | ❌ |
| 1,000,000 | ✅ ~0.01s | ✅ ~0.2s | ❌ | ❌ |

import sys

数据量 = int(原始数据[0])

# ✅ 五、边界条件检查清单

> **提交前花 30 秒过一遍，能避免 80% 的 WA。**

### 必查 6 题

| # | 问题 | 具体怎么测 |
|:---:|:---|:---|
| 1 | **n = 1 时**代码对吗？ | 手动输入最小规模，看输出是否合理 |
| 2 | **n = 最大时**会超时/溢出吗？ | 心算一下复杂度 × 最大 n 的量级 |
| 3 | **全是相同值**还能对吗？ | 试 `[5,5,5]` 或 `"aaaa"` |
| 4 | **负数/零**会出问题吗？ | 试 `[0, -1, -2]`，除法特别注意 |
| 5 | **空输入**（空串、空列表）？ | 试 `""` 或 `[]` |
| 6 | **答案要取模**（mod 10⁹+7）？ | 加上了吗？减法取模后加了 MOD 防负数吗？ |

### 提交前速查动作

```
□ 样例输出和题目一模一样吗？（空格、换行、小数位数）
□ n = 1 手动测过了吗？
□ 变量都初始化了吗？（尤其是计数的、累加的）
□ 多组输入时，每组之间清空状态了吗？
□ 输出格式：空格分隔用 print(*a)，不要末尾多空格
□ 整数除法用了 // 而不是 / 吗？
```

# 🐛 六、常见错误速查表

### 运行时错误（RE）

| 报错信息 | 最常见的原因 | 怎么修 |
|:---|:---|:---|
| `IndexError: list index out of range` | 数组下标越界 | 检查 `range()` 范围，数组多开几位 |
| `KeyError: 'xxx'` | 字典 key 不存在直接取了 | 换成 `字典.get(key, 默认值)` |
| `RecursionError` | DFS 递归太深 | 开头加 `sys.setrecursionlimit(1000000)` |
| `ZeroDivisionError` | 除法分母为 0 | 除法前加 `if b != 0` |
| `ValueError` | 输入有空行或格式不对 | 读入时加 `.strip()`，检查输入格式 |

### 答案错误（WA）

| 现象 | 可能原因 | 怎么修 |
|:---|:---|:---|
| 样例对了但交上去 WA | 没处理边界 | 跑「边界条件检查清单」 |
| 输出和答案长得一样但判 WA | 多了空格/少了换行 | 用 `print(*a)` 或 `" ".join()` 确保格式 |
| 结果不对但看起来差不多 | 用了 `/` 而不是 `//` | 整数运算全部 `//` |
| 原数组莫名其妙被改了 | `sort()` 改了原列表 | 不想改原列表用 `sorted()` |
| 循环里删元素结果少处理了 | 遍历中修改列表 | 倒序遍历或推导式新建 |

### 超时（TLE）

| 写法 | 为什么要命 | 怎么改 |
|:---|:---|:---|
| `list.pop(0)` | $O(n)$！每次删头都要搬所有元素 | 换成 `deque.popleft()` |
| `if x in list` 在循环里 | 每次 $O(n)$，里外相乘 $O(n^2)$ | 把 list 转成 set：`if x in set()` |
| `s += ch` 循环拼接字符串 | 每次创建新字符串，越拼越慢 | 用 list 收集，最后 `"".join()` |
| 数据 10⁵ 写了 $O(n^2)$ | 10¹⁰ 次操作，远超 1 秒 | 看 n 范围改算法（参考输入规模对照表） |

# 🔧 七、调试技巧

### 方法 1：print 大法（最实用）

```python
def solve():
    数据 = list(map(int, input().split()))
    print(f"DEBUG 读入数据: {数据}")               # 先确认输入对不对

    结果 = 核心处理(数据)
    print(f"DEBUG 中间结果: {结果}")               # 看中间计算是否正常

    print(结果)                                     # 最后输出答案
```

### 方法 2：二分定位

> 把代码一段段注释掉，看哪段注释后 bug 消失，就在那段里。

### 方法 3：缩到最小用例

> 如果大样例报错但你不知道为啥：把输入缩到 n=2~3，手算出正确答案，和你的输出对比。

### 心理提醒

- **AC 率 30%~40% 是完全正常的**。交 3 次对 1 次，不等于你菜。
- **15 分钟没头绪就看题解**。学长说的——不丢人，这是高效学习方法。
- **WA 了先理解为什么，再改代码**。别盲目试"改个值看看"。
- **一次只改一处**，改完就交，不然不知道哪个改动生效了。

# 🧠 八、你的思维模式升级指南

## 暴力 vs 数学 —— 什么时候该动脑，什么时候该动手

> 从你的代码里我看到了一个经典挣扎：你知道数学解法存在，但觉得"思维不清楚"，于是选择暴力模拟。这很正常，但需要升级。

```
决策树：

题目有明确的数学公式吗？
  ├─ 有（等差数列、组合数、整除特征）
  │     → 必须用数学！暴力大概率超时
  │       例：统计方形(C(n,2)×C(m,2))、贪心排队(排序后公式)
  │
  ├─ 有规律但你没找到
  │     → 先暴力跑 n=1~20 的小数据，打印结果找规律
  │       例：斐波那契递推、金币问题(每天给 i 枚共 i 天)
  │
  └─ 真没有公式，只能一步步模拟
        → 暴力就暴力，但注意复杂度
          例：不高兴的津津(按天模拟)、陶陶摘苹果(逐个判断)
```

**你常犯的模式**：小鱼的航程那道题，你写了数学推导又删了，理由是"思维不清楚"，最终暴力过了。

**怎么改**：
- 先暴力写出来（确保正确），跑小数据
- 打印中间结果，观察规律
- 如果有规律 → 改成公式，复杂度从 $O(n)$ 降到 $O(1)$
- 没规律 → 暴力已经是最优解了，不纠结

## 为什么你有时做不出来

看了你 28 份代码后，我总结了 5 个高频卡点：

| # | 卡点 | 你代码里的表现 | 怎么破 |
|:---:|:---|:---|:---|
| 1 | **索引混乱** | 车厢重组那道冒泡排序，`j` 和 `i` 搞混了，`nums[j]` 写成了 `nums[i]` | 嵌套循环前先在纸上写下：**外层代表什么、内层代表什么、每次循环的边界是什么** |
| 2 | **过早放弃数学思路** | 小鱼的航程删了数学推导、研究斐波那契直接未完成 | 数学思路卡住时，画具体数字例子（n=7 手工推一遍），不要凭空想 |
| 3 | **写完不测试边界** | 最长单词2 只处理了末尾单个句号 | 提交前强制跑「边界条件检查清单」（第五章），养成肌肉记忆 |
| 4 | **试错式编程** | 利润计算、出现次数超过一半 里有大量注释掉的旧代码 | 一种思路写到底，实在不行再全删重写。留着旧代码只会让思路更乱 |
| 5 | **函数使用不足** | 86% 的代码是全局平铺，递归阶乘、质数判断才被迫写函数 | 把独立逻辑抽成函数，出 bug 时一眼定位。不是"为了规范"，是"为了好调" |

## 做完题之后 —— 30 秒收尾习惯

> 你的代码里到处都是注释掉的旧尝试。这说明你在探索，但探索完之后没收尾。

```
□ 删掉所有注释掉的旧代码（git 会帮你记住的！）
□ 变量名看一眼：n、x、s 改不改？（本章第 4 行：没人逼你用短名字）
□ 如果同一道题在另一个平台也做了 → 删掉重复的，只留最优版
□ 超时的代码删掉，或者加 # TODO: 优化到 O(n log n)
```

# 🏆 九、蓝桥杯比赛策略

### 赛前准备

```
□ 模板文件放在手边：算法模板库.md 打开就能翻
□ Python 文件开头预写：
    import sys
    sys.setrecursionlimit(1000000)
    from collections import deque, Counter, defaultdict
    import math, bisect, heapq, itertools
□ 速查：语法手册.md Ctrl+F 查语法
```

### 赛中策略

```
1. 通读所有题（10 分钟）→ 标上「简单/中等/难」
2. 先秒最简单的那道，建立信心
3. 中档题按"会做"优先，不按题号顺序
4. 每题提交前 30 秒 → 跑边界检查清单
5. 一道题卡 20 分钟 → 果断跳过，回头再看
6. 最后 20 分钟 → 不再写新题，复查已过的代码
```

### 骗分保底

```
□ 不会做 ≠ 0 分！很多 OJ 有部分分（过了部分测试点就有分）
□ 暴力枚举可能过 30% 的数据点，聊胜于无
□ 特判 corner case（n=1、全是 0 等）可能白嫖一个测试点
□ 题目完全不会：读入数据后输出 0 / -1 / 样例答案，碰运气
```

### Python 组的特殊优势

```
□ 蓝桥杯 Python 组竞争远小于 C++ 组，获奖门槛更低
□ Python 写代码快 → 多出来的时间留给思考
□ 整数永不溢出、字符串操作方便 → 善用
□ 省一不需要全对，做出 2~3 道完整题 + 部分分就有希望
```

## 五个核心问题

### 问题 1：嵌套循环索引混乱 🔴

**证据**：车厢重组 (2117) — nums[j] 写成了 nums[i]

**根因**：用 i、j 这种没语义的单字母做变量名。循环套循环时 i 和 j 在脑子里打架。

**解法**：嵌套循环用中文变量名——「轮数」「当前位置」「行」「列」。名字会说人话，bug 少一半。

→ [[../10-语法手册/03-流程控制#嵌套循环易错点]]

---

### 问题 2：遇到数学思路就退缩 🔴

**证据**：小鱼的航程 — 写了数学推导又删了，注释"使用数学手法写得自己思维不清楚"；研究斐波那契 — 写着"未完成……"放弃。

**这不是能力问题，是信心问题。** 你不是不会数学，你是不确定自己的推导。

**解法**——下次遇到数学题，强制按这个流程：
1. 先在纸上写 n=1,2,3,4,5 的答案（手算）
2. 观察这 5 个答案的关系
3. 发现规律 → 写出公式 → 验证 n=6
4. 代码实现公式

→ [[../20-刷题工具箱/00-解题全流程]]

---

### 问题 3：写完不测试边界 🟡

**证据**：最长单词2 (2961) — `s.rstrip(".")` 只去掉末尾句号，如果句子是 "hello... world..." 只会去掉最后一个点。OJ 样例刚好只有一个句号所以 AC 了——换样例就 WA。

**解法**——提交前 30 秒：
- n=1 时，我的代码输出什么？
- 全是 0 / 全是相同值 / 全是负数？
- 空字符串 "" 会怎样？

→ [[../20-刷题工具箱/03-边界检查清单]]

---

### 问题 4：试错式编程，旧代码不清理 🟡

**证据**：利润计算 (1010) — 两个版本的解法，中间夹着注释掉的旧代码；2973 — 76 行文件里 40 行是注释掉的旧尝试。

**根因**：留着旧代码说明心里没底——不确定当前解法是不是最好的。

**解法**：AC 之后删掉所有注释掉的旧代码，只留最优版本。git 会帮你记住历史。

---

### 问题 5：几乎不写函数 🟢

**证据**：28 份代码中 24 份（86%）是全局平铺。但在 stats.py 里反而结构清晰——说明你「会」写函数，只是觉得刷题时「没必要」。

**为什么刷题也需要函数**：
- 出 bug 时一眼定位
- 质数判断写一次，三道题都能复用
- 主逻辑 3~5 行，脑子只装一件事

**最小成本方案**：每个题至少写 `def solve():` 把主逻辑包起来。

→ [[../10-语法手册/09-函数]]

## 已经被自己证明过的优势

### 自我反思型注释
你写的不是"这行干啥"的死注释，而是"我当时怎么想的"的活注释：
```
# 我这个题做的真棒，中间那个替换下一步的迭代相当棒
# 思考！累加变量 = 累加变量 + 当时值
# 神来之笔，语法糖
```
这种注释价值远大于 `# 遍历数组`。它记录的是**顿悟时刻**。

### Python 特性的自然运用
`for-else`、`max(key=len)`、`dict.get(key, 0)`、列表推导式——已经越过了"翻译 C++ 到 Python"的阶段。

### 工具意识
`stats.py` 是非常用心的个人刷题追踪工具。CLI 参数、git 集成、README 自动更新——不是刷题机器会做的事。

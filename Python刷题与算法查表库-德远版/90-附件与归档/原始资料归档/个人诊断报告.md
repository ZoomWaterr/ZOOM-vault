## 五个核心问题（按严重程度排序）

### 问题 1：嵌套循环索引混乱

**你代码里的证据**：
```
车厢重组（冒泡排序）：nums[j] 写成了 nums[i]
```
`i` 是外层轮数，`j` 是内层当前位置——你把它们搞混了，导致排序逻辑完全错误。

**为什么会这样**：
你用 `i` 和 `j` 这种没有语义的单字母做变量名。当循环套循环时，`i` 和 `j` 在脑子里打架。

**怎么改**：
```
把 for i in range(n)      改成  for 轮数 in range(n)
把 for j in range(n-1-i)  改成  for 当前位置 in range(n - 1 - 轮数)
```
名字本身就在告诉你它代表什么。你看到 `if 数组[轮数] > 数组[轮数+1]` 立刻就知道不对——轮数怎么能跟轮数+1 比？应该是 `数组[当前位置] > 数组[当前位置+1]`。

**立刻行动**：以后写嵌套循环时，外层用「轮数」「行」「天数」，内层用「当前位置」「列」「同学编号」。强迫自己不用 `i` `j`。

### 问题 2：遇到数学思路就退缩

**你代码里的证据**：
```
小鱼的航程：写了数学推导，删掉，注释"使用数学手法写得自己思维不清楚，
不如下面的直接暴力"
研究斐波那契：文件末写着 #未完成......
```

**这不是能力问题，是信心问题。** 你显然知道数学解法存在——你甚至尝试过推导。但你一遇到"思维不清楚"的感觉就立刻退回舒适区（暴力模拟）。

**为什么数学思路会"不清楚"**：
1. 你在脑子里空想，没有在纸上画
2. 你试图一步跳到最终公式，没有先从 n=1,2,3 的小例子推起
3. 你把"数学"当成天才才用的东西，觉得自己不够格

**怎么改**——下次遇到需要数学的题，强制自己按这个流程走：

```
1. 先在纸上写出 n=1,2,3,4,5 的答案（手算，不是写代码）
2. 观察这 5 个答案之间有什么关系（差是常数？倍数是常数？）
3. 如果发现规律 → 写出公式 → 验证 n=6 手算和公式是否一致
4. 一致 → 代码实现公式（O(1) 或 O(n)）
5. 不一致 → 说明规律找错了，但至少你尝试了，比你直接暴力要学到更多
```

**具体例子**——你做过的那道"统计方形"：
- 正方形数 = 对每个边长 k 算 `(n-k+1) * (m-k+1)` 累加
- 总矩形数 = `C(n+1,2) * C(m+1,2)` = `n*(n+1)/2 * m*(m+1)/2`
- 如果直接暴力数 n=5000,m=5000 会超时，所以**必须**用公式

这就是数学解法的威力——把 $O(n)$ 降到 $O(1)$。

### 问题 3：写完不测试边界，依赖 OJ 判定

**你代码里的证据**：
```
最长单词2：s.rstrip(".") 只去掉末尾一个句号，如果句子是 "hello... world..."
          只会去掉最后一个点，不是全部点——但 OJ 的样例恰好只有一个句号，
          所以 AC 了。换个样例就 WA。
平台优惠：total -= num[2:lens:3] —— 列表切片减整数，类型错误，根本跑不了
```

**为什么这很危险**：
OJ 的测试用例是有限的。样例过了 ≠ 代码对了。蓝桥杯正式比赛的测试用例比你想象的丰富，靠"交了碰运气"会死得很惨。

**怎么改**——提交前 30 秒，逼自己做这三件事：

```
1. n=1 时，我的代码输出什么？（手动跑一遍）
2. 输入全是 0 / 全是相同值 / 全是负数，我的代码还能对吗？
3. 如果问题涉及字符串，空字符串 "" 会怎样？
```

我把这个做成了清单放在刷题方法论第五章——但清单只是辅助，习惯得靠你自己养。

### 问题 4：试错式编程，旧代码不清理

**你代码里的证据**：

```
利润计算(1010)：两个版本的解法（自己写的 + "大佬写的"），中间夹着注释掉的旧代码
出现次数超过一半(2973)：76 行文件，40 行是空行和注释掉的旧尝试
级数求和(P1035)：注释掉的 for 循环和 while 循环各一份
```

**为什么这是个问题**：
1. 你自己回头看代码时，不知道哪段是"活"的、哪段是"死"的
2. 想复习时，注意力被旧代码分散
3. **留着旧代码说明你心里没底**——不确定当前的解法是不是最好的

**怎么改**——养成一个 5 秒习惯：
```
AC 之后 → 删掉所有注释掉的旧代码 → 只留最优版本 → commit
```
git 会帮你记住历史的。代码文件只保留"当前最优解"。

例外：如果旧解法展示了不同的思路（暴力 vs 数学），**留一份并加注释说明"这是 O(n²) 暴力版，超时但思路清晰"**。

### 问题 5：几乎不写函数，全部平铺

**你代码里的证据**：
```
28 份代码中，24 份（86%）是全局平铺的脚本
只有递归阶乘、质数判断、回文判断这三个场景被迫写了函数
stats.py 是你自己写的工具，反而结构清晰——说明你"会"写函数，
只是觉得刷题时"没必要"
```

**为什么刷题也需要函数**：
1. **出 bug 时定位更快**：如果质数判断逻辑有错，你知道去看 `def 是素数吗(n)` 而不是在一大坨平铺代码里翻
2. **复用**：质数判断在 P1217、P5723、P1022 三道题里都需要——写一次函数，三次复制
3. **省脑容量**：做完 `def 读入数据()` 和 `def 输出结果()`，主逻辑只需要 5 行，脑子不用同时装所有细节

```python
# 每个题至少写这两个函数：
def 读入():
    return map(int, input().split())

def 核心逻辑(数据):
    # 你的算法在这里
    return 答案

if __name__ == "__main__":
    数据 = 读入()
    print(核心逻辑(数据))
```

## 你已经被自己证明过的优势

说完问题，也要说你做对了什么。这些你不用改，继续保持：

### 自我反思型注释
你写的不是"这行干啥"的死注释，而是"我当时怎么想的"的活注释：
```
# 我这个题做的真棒，中间那个替换下一步的迭代相当棒
# 思考！累加变量 = 累加变量 + 当时值
# 神来之笔，语法糖
```
这种注释的价值远大于 `# 遍历数组`。它记录的是**顿悟时刻**——下次回来看，你能重新激活当时的思维路径。

### Python 特性的自然运用
`for-else`、`max(key=len)`、`dict.get(key, 0)`、列表推导式——这些不是每个初学者都会用的。你已经越过了"翻译 C++ 代码到 Python"的阶段，开始用 Python 的方式写 Python。

# 从题目到算法的思考流程

你做题时最需要形成一条固定流程。不是看到题就立刻敲代码，而是先把题目翻译成模型。

## 总流程

```text
读题
  ↓
找输入规模
  ↓
写暴力想法
  ↓
估复杂度
  ↓
找重复计算或结构特征
  ↓
选择模型
  ↓
写变量含义
  ↓
写代码
  ↓
测边界
  ↓
复盘
```

## 第一步：先找输入规模

数据范围决定你能不能暴力。

| 数据范围 | 大概能接受的做法 |
|---|---|
| `n <= 100` | `O(n^2)` 常常可以 |
| `n <= 1000` | `O(n^2)` 勉强看题 |
| `n <= 10^5` | 多半要 `O(n)` 或 `O(n log n)` |
| `n <= 20` | 可能是回溯、状态压缩 |
| `n` 很大但有单调性 | 可能是二分答案 |

## 第二步：暴力不是丢人，是起点

先问：

> 如果我不优化，最直接怎么做？

你必须能写出暴力，才知道优化在优化什么。

## 第三步：找题目里的结构

| 结构 | 对应模型 |
|---|---|
| 出现次数 | [[哈希计数]] |
| 连续一段 | [[前缀和]] / [[滑动窗口]] |
| 区间批量修改 | [[差分]] |
| 有序 | [[二分查找]] / [[双指针]] |
| 所有可能选择 | [[回溯]] |
| 最短步数 | [[BFS]] |
| 当前依赖前面 | [[动态规划]] |

## 第四步：写变量含义

写代码前先写：

```python
# i 表示：
# j 表示：
# cnt 表示：
# ans 表示：
```

你现在很多模糊感，其实来自变量角色没定清楚。

## 第五步：检查边界

至少测：

- 最小输入。
- 最大输入的复杂度。
- 全相同。
- 全不同。
- 只有一个元素。
- 下标刚好在边界。

## 真正的提升点

每题都按这条流程走，做 30 道以后，你脑子里会自然出现“题目 -> 模型”的路线。
